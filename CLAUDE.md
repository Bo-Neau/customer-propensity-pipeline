# Project Context: Propensity Pipeline (PySpark + Airflow rebuild)

## What this is
Rebuilding an existing pandas/XGBoost customer-propensity-to-purchase
pipeline as a distributed, orchestrated pipeline. PySpark handles ETL and
feature engineering, Apache Airflow 3 (Docker) handles orchestration, GX
Core is the data quality gate, XGBoost plus MLflow handle training and the
model registry. Dataset: REES46 e-commerce behavior data (Kaggle), one
month file at a time (40 to 67 million rows).

The goal of this rebuild is the orchestration and distributed-processing
layer, not the model itself. Keep the XGBoost hyperparameters and the
general modelling approach close to what a pandas pipeline would already
do, the README claim is "migrated to a distributed, orchestrated
architecture," not "built a better model."

## Verify these FIRST, before running anything
This scaffold was generated by Claude in chat without access to a live
Docker daemon, three things are flagged as uncertain and should be the
first things checked, ideally by fetching the current docs rather than
trusting this comment:

1. **Airflow 3 docker-compose.yaml, the auth/init step.** `airflow-init`
   currently only runs `airflow db migrate`. Airflow 3's default auth
   manager generates its own admin credentials rather than using
   `airflow users create` (that command is from the older FAB-based auth
   manager, which is no longer the default). If login to the UI fails,
   check `docker compose logs airflow-api-server` for an auto-generated
   password, or fetch
   https://airflow.apache.org/docs/apache-airflow/stable/howto/docker-compose/index.html
   for the current pattern.
2. **DAG import path.** `dags/propensity_dag.py` tries
   `from airflow.sdk import dag, task` first and falls back to
   `from airflow.decorators import dag, task`. Confirm which one is
   actually correct for the installed Airflow 3.2.x against
   https://airflow.apache.org/docs/apache-airflow/stable/core-concepts/taskflow.html
   and simplify the DAG file to just the right import once confirmed.
3. **GX Core API in src/quality_gate.py.** Uses the newer Fluent/Core API
   (`gx.get_context`, `add_dataframe_asset`, `add_batch_definition_whole_dataframe`).
   GX Core's Python API has changed across versions. Confirm the exact
   method names against https://docs.greatexpectations.io and fix any
   signature mismatches, the overall structure (build a suite, validate,
   raise on failure) should still be right even if a method name moved.

## Commands
- Build and start: `docker compose up --build`
- Tail one service: `docker compose logs -f airflow-scheduler`
- Trigger the DAG from the CLI: `docker compose exec airflow-scheduler airflow dags trigger propensity_pipeline`
- Run the fast unit test: `pytest tests/`
- Tear down clean (use this if anything gets into a weird state): `docker compose down --volumes --remove-orphans`

## Architecture map
- `dags/propensity_dag.py`: the DAG, wires the 7 tasks together with TaskFlow
- `src/ingest.py`: raw CSV to staged Parquet (PySpark)
- `src/validate_schema.py`: structural checks (PySpark), raises on missing columns or empty data
- `src/transform.py`: feature engineering, aggregates raw events to one row per user_session (PySpark)
- `src/quality_gate.py`: semantic checks on the feature table (GX Core), raises on failed expectations
- `src/train.py`: bridges to pandas, trains XGBoost, logs to MLflow
- `src/evaluate.py`: AUC threshold gate against the MLflow run
- `src/deploy.py`: promotes the MLflow model version to Production
- `tests/test_transform.py`: fast smoke test against synthetic data, run this before touching Airflow at all

## Working rules
- Never pass a DataFrame through Airflow XCom, only file paths or small strings (a run_id, a status string).
- Don't introduce a separate Spark cluster (spark-master/spark-worker containers) unless explicitly asked, PySpark runs in local[*] mode inside the Airflow worker for this project. This was a deliberate scope decision to avoid extra infrastructure.
- Keep `requirements.txt` and `docker/Dockerfile` in sync, anything added to one without the other will fail silently on the next rebuild.
- No em dashes or en dashes in any committed documentation, comments, or commit messages, use periods or commas instead.
- This is a portfolio project for a job search, prioritize a working, demonstrable pipeline (green DAG run, a real screenshot) over chasing perfect distributed-training purity. The `.toPandas()` bridge in train.py is intentional, not a shortcut to fix later.
